# ADR-008: File Storage Provider — Deferred to Post-V1

## Status
Deferred

> **DEFERRED**: NST-Events V1 will not support file uploads. All avatar, banner, logo, certificate, and media upload features are deferred to a later release (V1.1 / V2). V1 will use default or generated fallback assets only. File storage provider selection is **not a blocker** for starting backend implementation.

---

## Context
The platform will eventually need object storage for:
- User avatars
- Event banner images
- Club banner images
- Secure documents (budget proposals, faculty sign-offs) — private access only

Media must not be stored in PostgreSQL columns. When implemented, the Express backend must mediate all access — clients never receive raw storage credentials.

---

## V1 Decision
**File uploads are not part of V1.**

The team evaluated the following options but determined that selecting a storage provider is not required for the initial product launch. V1 will use default/generated fallback assets (e.g., initials-based avatars, placeholder banners). All database fields that would reference uploaded media (`avatar_url`, `banner_url`, etc.) remain nullable in the schema and will store `NULL` in V1.

This decision **unblocks backend implementation immediately**. The storage provider will be selected when file upload features are prioritized in a future release.

---

## Options Preserved for Future Selection

| Option | Pros | Cons |
|---|---|---|
| **Cloudflare R2** | Zero egress fees, S3-compatible SDK, global CDN via Cloudflare | Requires Cloudflare account and domain, not self-hosted |
| **AWS S3** | Industry standard, excellent Node.js SDK, widely documented | Egress costs, external dependency, not self-hosted |
| **MinIO on NST Cluster** | Fully self-hosted, S3-compatible, no external dependency, free | Requires persistent storage on K3s (StatefulSet + PVC), operational overhead |

## Consequences (Applicable When Implemented — Not V1)
- Express backend handles all upload authorization (pre-signed URLs) — clients never access storage with raw credentials
- File validation (MIME type, size limits) is enforced by Express before any URL is issued
- Public buckets (`avatars`, `event_media`, `club_banners`) served via CDN or public URL
- Private bucket (`secure_documents`) requires signed read URL generated by Express

---

## What Has Been Ruled Out
- **PostgreSQL BLOBs**: Rejected — severe performance degradation
- **Supabase Storage**: Rejected along with the Supabase platform
